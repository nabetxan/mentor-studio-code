# Phase 2.5: ルールファイル改善 Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Restructure the mentor rule files to be more robust when plan/spec files don't follow expected formats, and improve the template system so setup generates better-structured starter files.

**Architecture:** Current pain points: (1) MENTOR_RULES.md references `app-design.md` and `learning-roadmap.md` but these might not exist or might have a totally different structure. (2) Task completion logic in MENTOR_SKILL.md assumes roadmap has `⬜`/`✅` markers and specific section structure. (3) The setup command generates template files as string literals in `mentorFiles.ts` which is hard to maintain. Solution: Add graceful handling for missing/unstructured plan files, improve starter templates with clear section markers, and restructure the template system for maintainability.

**Tech Stack:** TypeScript, VSCode Extension API, Vitest

---

## Background: Current Problems

### Problem 1: Plan/Spec format assumptions

`MENTOR_SKILL.md` assumes:
- `learning-roadmap.md` has a task table with `⬜`/`✅` checkboxes
- `learning-roadmap.md` has `### ✅ 完了の定義` sections per task
- `app-design.md` exists and has specific structure

If a user provides their own plan/spec files (or none at all), these assumptions break and the mentor AI gets confused.

### Problem 2: Rigid task completion flow

The task completion instructions say "change `⬜` → `✅`" and "check off items in `### ✅ 完了の定義`" — but if the user's roadmap doesn't use this format, the mentor either fails silently or edits the file incorrectly.

### Problem 3: Template maintainability

All template content is in one large `mentorFiles.ts` file as template literals. Adding/modifying rules requires editing escaped strings. Error-prone and hard to review.

## Chunk 2: Improve MENTOR_SKILL.md for Flexible Plan/Spec Handling

### Task 3: Add task-management rules file

**Files:**
- Create: `extension/src/templates/rules/task-management.md.ts`

MENTOR_SKILL.md references `docs/mentor/rules/task-management.md` but the setup command doesn't create it. This task adds it.

- [ ] **Step 1: Create task-management template**

```typescript
export const content = `# Task Management Rules

## Task Completion Checklist

When a user completes a task, do ALL of these **immediately**:

### 1. Update progress.json

\`\`\`
completed_tasks: add { task, name, plan }
current_task: increment to next task number
next_suggest: description of next task
resume_context: 1-2 sentence summary of what was accomplished
\`\`\`

### 2. Update current-task.md

- Read the roadmap/plan file to find the next task content
- If the roadmap has structured tasks (numbered sections, checkboxes), copy the next task content
- If the roadmap does NOT have structured tasks, write a brief "Next: [topic]" placeholder and ask the user what to work on next
- Overwrite \`docs/mentor/current-task.md\` with the next task

### 3. Update the roadmap (if it has task markers)

Only do this if the roadmap uses recognizable task markers. Common formats:

| Format | Action |
|--------|--------|
| \`⬜\` / \`✅\` checkboxes | Change \`⬜\` → \`✅\` for completed task |
| \`- [ ]\` / \`- [x]\` markdown checkboxes | Change \`[ ]\` → \`[x]\` |
| Numbered task sections with completion criteria | Check off completion items |
| No task markers | Skip roadmap update, just update progress.json |

**Key rule:** If the roadmap format is unclear, do NOT guess. Ask the user: "Your roadmap doesn't have a standard task format — should I add checkboxes, or just track completion in progress.json?"

### 4. Record any learning gaps

If the user answered any questions incorrectly during this task:
- Ensure they are in \`unresolved_gaps\` in progress.json
- Ensure they are in question-history.json with \`correct: false\`

## Task Start

When starting a new task:

1. Read \`current-task.md\` for the task content
2. Check \`unresolved_gaps\` — if any match this task's topic, propose review first
3. If \`current-task.md\` is empty or says "No task assigned", read the roadmap to find the next task
4. If no roadmap exists, ask the user what they want to work on
`;
```

- [ ] **Step 1: Update the template**

Key changes to MENTOR_SKILL.md content:

**Replace the rigid "On Task Completion" section** with:

```markdown
## On Task Completion (Critical)

When user completes a task, **immediately** do ALL of these:

1. **Update progress.json** (always required):
   - Add task to `completed_tasks`
   - Increment `current_task`
   - Update `next_suggest` and `resume_context`

2. **Update current-task.md** (always required):
   - Load the roadmap/plan file
   - Find and copy next task content → overwrite current-task.md
   - If roadmap has no clear next task, ask the user

3. **Update roadmap** (only if it has task markers):
   - See `docs/mentor/rules/task-management.md` for supported formats
   - If format is unclear, ask user before modifying

**See**: `docs/mentor/rules/task-management.md` for detailed rules
```

**Replace the rigid "Intake + Plan" section** with:

```markdown
## Working With Plan/Spec Files

The user may provide plan and spec files in various formats:

### Structured plans (roadmap with tasks)
- Has numbered tasks, checkboxes, or clear sections
- Use task markers for completion tracking
- Follow the task order unless user requests otherwise

### Unstructured plans (goals doc, feature list, etc.)
- No clear task breakdown
- Use as context/reference, but derive tasks from discussion with user
- Track tasks solely via progress.json

### No plan files
- Run intake to create initial plan
- See intake rules below

### App design / spec files
- May be detailed (with screens, data models) or high-level (goals only)
- Use whatever detail level is available
- Don't assume specific sections exist — check first

## Intake + Plan (Only If Needed)

Trigger intake only when:
- `current-task.md` does not exist, OR
- progress.json does not describe a clear next task, OR
- The user has no project plan/docs yet

When intake is required:
1. Ask about the app idea/goals, target platform/stack, and prior knowledge
2. Confirm the project scope and timeline
3. Create a learning plan and the first task
4. Update `progress.json` to reflect the new plan
```
